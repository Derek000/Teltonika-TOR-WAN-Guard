# Architecture

## Design objective

Protect WAN-exposed services from new inbound traffic originating from Tor exit nodes, while preserving outbound LAN-origin Tor usage for legitimate OSINT, penetration testing, research, and privacy-preserving workflows.

## Traffic policy

| Flow | Action | Reason |
|---|---:|---|
| LAN -> WAN | Allow | Supports normal browsing, OSINT, authorised testing |
| LAN -> Tor | Allow | Supports legitimate OSINT workflows |
| WAN -> Router from Tor exit | Drop | Protects router services |
| WAN -> Port-forwarded service from Tor exit | Drop | Protects internal exposed services |
| Established/related return traffic | Allow | Preserves stateful outbound connectivity |

## Chain usage

- `INPUT`: traffic destined to the router itself.
- `FORWARD`: traffic traversing the router, including DNAT/port-forwarded services.
- `OUTPUT`: traffic generated by the router itself. This project does not block router-origin traffic.

## Stateful control

The key control is:

```sh
-m conntrack --ctstate NEW
```

This means only new inbound sessions from Tor exit nodes are dropped. Return traffic for LAN-origin sessions should be classified as `ESTABLISHED` or `RELATED` and allowed.

## Interface scoping

The rules are deliberately scoped to the WAN interface:

```sh
-i "$WAN_IFACE"
```

This avoids blocking LAN-origin traffic and avoids unexpected east-west LAN effects.
